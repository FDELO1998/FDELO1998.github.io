    ###Context
    context的应用场景官方的文档说的是，共享对于一个‘组件树’来说是全局的数据。通过context可以避免中间元素传递props

    使用context的好处就是，减少数据逐层传递的冗杂，但是缺点是，context可以理解为是创建了全局的变量。如果使用过多，代码就会比较混乱。

    redux(connect)

    redux 通用状态管理。我们知道在react中将组件分为UI组件和容器组件。前者只负责界面呈现，不带任何业务逻辑，没有状态，所有数据都由参数提供。后者负责管理数据和业务逻辑。
    React-Redux 提供connect方法，用于从 UI 组件生成容器组件

    这是最近我工作中的一个小例子，MiniProgramList是我的UI组件 导出的就是通过connect方法自动生成的容器组件。

    当然我们还需要业务逻辑，也就是MiniProgramList这个组件需要输入逻辑和输出逻辑。

    （1）输入逻辑：外部的数据（即state对象）如何转换为 UI 组件的参数

    （2）输出逻辑：用户发出的动作如何变为 Action 对象，从 UI 组件传出去。

    所以connect的完整API是

    mapStateToProps()

    所以我上面的mapStateToProps就是mapStateToProps，Object.assign({},CodeManageActions,)就是mapDispatchToProps它们定义了 UI 组件的业务逻辑。前者负责输入逻辑，即将state映射到 UI 组件的参数（props），后者负责输出逻辑，即将用户对 UI 组件的操作映射成 Action。
    mapStateToProps是建立一个外部的）state对象到（UI 组件的）props对象的映射关系。就是拿到存储在Store中的值，这个值是通过 组件中的state传下来的，我们所有的组件都是包裹在Provider下面的。

    mapStateToProps会订阅 Store，这样每当state更新的时候，都会触发UI组件的重新渲染。
    （个人理解：mapStateToProps就是订阅state里的参数，比如我的示例中{ codeManage }是最外层store中的一个值，当store中的它发生变化，那么本组件就会触发重新渲染）

    mapDispatchToProps()

    就是第二个参数，用来建立 UI 组件的参数到store.dispatch方法的映射。也就是说，它定义了哪些用户的操作应该当作 Action，传给 Store。它可以是一个函数，也可以是一个对象。在我上面的例子中CodeManageActions是请求接口返回的数据的模块

    可以看到在这里，定义了Action 以及怎样发出Action。定义了请求接口的函数，然后将返回的值赋给创建的action，这里sdkList是我们要用在多个组件的值，也就是redux里的值。创建了action receiveSdkList去修改它。这样redux中的sdkList就会改变。

    Hooks

    react v16.8引入了全新的API Hooks，颠覆了之前的用法。
    主要通过函数组件+Hooks的方法去实现之前功能。函数组件与类组件的区分，网上很多。主要是没有state,没有生命周期，接受props，渲染dom不关注其他逻辑

    大概使用就是使用useState给原本state中的变量去创建状态，使用 useEffect()等钩子函数去实现之前生命周期的逻辑。hook是个庞大的部分。准备再整理一篇详细的使用。
